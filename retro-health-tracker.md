## Health Indicators

Review these every 3 sprints:

### Action Completion Rate
```
Actions Completed / Actions Generated x 100 = Completion Rate %

Target: > 70%
Warning: < 50% — retro actions are not being taken seriously
```

### Attendance Rate
```
Actual Attendees / Expected Attendees x 100 = Attendance Rate %

Target: 100% (Scrum team), PO optional
Warning: < 80% — retro is not being prioritised
```

### Team Energy Score
Ask each team member: *"Rate this retro from 1–5 — did it feel valuable?"*
Average the scores.

```
Target: > 3.5 consistently
Warning: < 3 for 2+ sprints — format or facilitation needs to change
```

## Retro Quality Checklist

Run this after every retro:

- [ ] Did we open by reviewing last sprint's action items?
- [ ] Did every team member contribute at least one sticky note?
- [ ] Did the team vote on priorities (not just list everything)?
- [ ] Did every action item get a named owner?
- [ ] Did every action item get a due date or target sprint?
- [ ] Did we close within the timebox?
- [ ] Did the team leave with a clear sense of what changes next sprint?

**Score: 7/7** — Excellent retro
**Score: 5–6/7** — Good, one area to improve
**Score: < 5/7** — Retro needs structural change

## Warning Signs Your Retros Are Broken

| Signal | What it means | Fix |
|--------|--------------|-----|
| Same issues appear every sprint | Actions aren't being completed or are too vague | Make actions smaller and more specific |
| Team goes quiet during retro | Psychological safety issue or format fatigue | Switch format, consider anonymous input |
| SM talks more than the team | Facilitation drift | SM goes silent — let the team fill the space |
| Retro runs over every time | Poor timeboxing | Use a visible timer, hard stop at 60 minutes |
| PO never attends | Retro seen as "dev team only" | Invite PO explicitly, frame it as delivery improvement |
| Actions assigned to "the team" | No real ownership | Every action must have one named person |
